# Tender Bot — работа с GigaChat

Проект для автоматизации обработки тендерной документации с использованием GigaChat (LLM от Сбера).  
Позволяет анализировать тексты, извлекать ключевые требования и формировать ответы.

---

## 📦 Требования

- Python 3.9+
- Git
- Токен доступа к GigaChat API

---

## 🚀 Быстрый старт

**1. Клонируйте репозиторий**

```bash
git clone https://github.com/sanorin/tenders.git
cd tenders
```
**2. Создайте и активируйте виртуальное окружение**
```bash
python -m venv venv

# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate
```
**3. Установите зависимости**
```bash
pip install -r requirements.txt
```
**4. Настройте переменные окружения**
```bash
cp .env.example .env
GIGACHAT_TOKEN=ваш_токен_сюда (заменить токен)
```
**5. Запустите проект**
```bash
python main.py
```
## Сообщение команде
1. Никогда не работайте напрямую в ветке main
Основная ветка — это стабильная версия. Все изменения делайте только через отдельные ветки. Прямые пуши в main запрещены.

2. Перед началом любой задачи обновите локальную main
Переключитесь на main и скачайте последние изменения:
`git checkout main`
`git pull origin main`
Если этого не сделать, вы будете отталкиваться от устаревшей версии и получите конфликты.

3. Создайте свою ветку для задачи
Название ветки должно быть осмысленным, например `feature/add-parser` или `fix/readme-typo`.
`git checkout -b feature/название-ветки`
Теперь вы в своей ветке — здесь можно делать любые изменения.

4. Работайте, коммитьте, но без мусора
Коммиты делайте маленькими и с понятными сообщениями.
Никогда не коммитьте: `.idea`, `.env`, `__pycache__`. Они уже в `.gitignore` — проверьте, что у вас они не появляются в git status.

5. Когда задача готова, отправьте ветку на GitHub
`git push origin feature/название-ветки`
Если ветки на GitHub ещё нет, она создастся автоматически.

6. Откройте Pull Request (PR)
На странице репозитория нажмите «Compare & pull request».
Выберите: ваша ветка → main.
Напишите краткое описание: что сделано, что проверять.
Запросите код-ревью у коллег.

7. Если Git говорит, что есть конфликты с main
Не пугайтесь. Конфликты — это нормально.
Переключитесь на свою ветку и втяните свежий main:
`git pull origin main`
Git покажет, какие файлы конфликтуют. Откройте их — внутри будут маркеры <<<<<<<, =======, >>>>>>>.
Уберите эти маркеры, оставьте нужный код (ваш или чужой, или объедините).
Сохраните файлы, затем:
`git add .`
`git commit -m "Resolve merge conflicts"`
`git push`
После этого конфликт в PR исчезнет.

8. После того как PR одобрен и слит (merged)
Можно удалить свою ветку — на GitHub появится кнопка «Delete branch».
Локально ветку тоже удалите, когда она больше не нужна:
`git checkout main`
`git pull origin main`
`git branch -d feature/название-ветки`

9. Запрещённые действия

   `git push --force` в общие ветки (main и любые другие, где работают коллеги) — это перетирает историю и ломает Git у всех.

    Прямой push в main без Pull Request.

    Коммит файлов с секретами (токенами, паролями) — используйте `.env` и никогда не загружайте его в репозиторий.

10. Что делать, если случайно закоммитили что-то лишнее
Сразу же:

    Добавьте этот файл/папку в `.gitignore`.

    Удалите из индекса Git: `git rm -r --cached .idea .env` (или конкретные файлы).

    Закоммитьте это удаление и запушьте.

    Если секрет уже попал в историю — срочно смените токен/пароль.

Главное правило
Если вы не уверены в команде — спросите. Лучше потратить минуту на вопрос, чем час на восстановление сломанного репозитория.
